# AGENTS_INSTRUCTION

This file is the operational guide for coding agents working in this repository.
Use it as the source of truth for how to implement ORM and Alembic changes safely.

## Mission

Build and maintain a reusable FastAPI + PostgreSQL base for microservices with:
- SQLAlchemy 2.x ORM
- Alembic migrations
- predictable, reversible schema evolution

## Non-Negotiable Rules

1. Keep this repo generic for cross-team reuse unless explicitly asked to add a service-specific PoC.
2. Never add cross-service foreign keys or shared tables.
3. Every schema change must have an Alembic migration.
4. Autogenerated migrations must be manually reviewed and adjusted for safety.
5. Prefer backward-compatible migrations (expand/contract).
6. Do not perform destructive schema changes in a single risky step.

## Environment and Runtime Notes

- `DATABASE_URL` is read from environment in `app/db/config.py`.
- `.env` is not auto-loaded by Alembic unless explicitly sourced in shell or loaded by code.
- Default local DB URL should remain valid for local Docker Postgres usage.

## Standard Implementation Workflow

When changing schema or models, follow this exact order:

1. Update SQLAlchemy models under `app/db/models/`.
2. Generate migration:
   - `make db-revision message="your change"`
3. Review migration in `alembic/versions/` and fix unsafe operations.
4. Apply migration:
   - `make db-upgrade`
5. Validate revision state:
   - `make db-current`
   - `make db-history`
6. Validate rollback path:
   - `make db-downgrade step=-1`
   - `make db-upgrade`
7. Validate no drift:
   - `make db-drift-check`

## Migration Safety Patterns

### Adding a new nullable column
- Safe in one step.

### Adding a non-null column to existing table
- Use temporary `server_default` in migration to backfill existing rows.
- Remove default in same migration if runtime default is not desired.

### Renames / type changes / drops
- Use expand/contract over multiple releases:
  1. Add new field/table/index
  2. Write to both old and new if needed
  3. Backfill in batches
  4. Switch reads
  5. Drop old field/table in later release

## Docker-Backed Validation (Required for local testing)

1. Start Postgres:
   - `docker compose up -d postgres`
2. Run migrations:
   - `make db-upgrade`
3. Start API:
   - `uvicorn app.main:app --reload --port 8000`
4. Verify DB objects and data using container:
   - `docker exec web-search-agent-postgres psql -U postgres -d web_search_agent -c "SELECT tablename FROM pg_tables WHERE schemaname='public';"`

## Definition of Done for Schema PRs

- Models updated and consistent with naming conventions.
- Migration file exists and is manually reviewed.
- Upgrade and downgrade tested locally.
- Drift check passes.
- Existing data remains intact after migration.
- Notes include rollback and lock-risk considerations.
